# 传输层

> **考情提示**：传输层是 408 网络分值第二高的章节——TCP 连接管理（三次握手/四次挥手）和 TCP 拥塞控制（慢开始→拥塞避免→快重传→快恢复）的窗口变化曲线图几乎每年必考。通常 1–2 道选择题（2–4 分）加一道大题中的子问题（5–8 分）。

## 传输层概述

### 传输层的功能

传输层 🔥 是网络协议栈中第一个提供**端到端（进程到进程）** 逻辑通信的层次——网络层只负责把分组投送到目标主机，但具体交给哪个应用程序（哪个进程）由传输层根据**端口号**区分。传输层的核心功能有三：

- **端到端逻辑通信**：在源进程和目的进程之间提供可靠/不可靠的数据传输通道——应用只管在两边的传输层通信。
- **复用和分用** 🔥：发送端（复用）——多个应用进程的数据通过不同的端口号汇聚到同一个 IP 层发出。接收端（分用）——网络层把分组交付给传输层后，传输层根据目的端口号投递到正确的应用进程。
- **差错检测**：利用报文段中的校验和对整段传输数据做完整性检验（不但检测首部错误，也检测数据部分错误——比 IP 层仅检首部的提升）。

### 套接字（Socket）

**套接字 = IP 地址 + 端口号** 🔥。在网络中唯一标识一个通信端点（一个网络进程）。一对套接字（源 IP:端口, 目的 IP:端口）构成一条传输层连接（TCP）或一次通信的标识（UDP）。端口号范围 0–65535，分为**熟知端口**（0–1023，如 HTTP 80、DNS 53）、**注册端口**（1024–49151）和**临时端口**（49152–65535，客户端动态分配）。

## UDP 协议

### UDP 的特点

UDP（User Datagram Protocol）🔥 是一种极简的传输层协议——仅在 IP 层基础上增加了**端口号**和**校验和**，其他一概不负责。总结特点：

- **无连接**：发送 UDP 数据报时无需先建立连接——直接打包发送。
- **不可靠（尽力而为）**：不保证数据一定到达、不保证顺序、不保证不重复。
- **面向报文**：发送方一次 send 发出一个完整报文，UDP 直接加上自己的首部后投递——不会拆分/合并应用层报文。
- **没有拥塞控制** 🔥：不管网络多负载，UDP 按照发送方的节奏全速发送——在网络拥塞时不退让。这正是多媒体直播应用选择 UDP 的核心原因——宁可丢几帧也不要卡顿。
- **首部长极短——仅 8 字节**（对比 TCP 的最小 20 字节）——大幅降低头部开销。
- **支持多播和广播**：一对一、一对多、多对多等任意交互模式——这一点 TCP 做不到（TCP 只一对一）。

### UDP 的应用场景

UDP 🔥 最适于对**实时性敏感但允许偶尔丢包**的场景：
- **DNS 域名查询**：单次请求-响应流量极小，建 TCP 连接的额外往返浪费不合算。
- **RIP 路由协议**：路由更新定时传播——依赖周期性重传而非可靠性保障。
- **多媒体通信**：直播、VoIP、在线游戏 🔥——丢一帧画面或几十毫秒音频比 TCP 重传导致的卡顿可忍受得多。
- **SNMP 网络管理协议**：少量数据查询和报警。

## TCP 协议

### TCP 的特点

TCP（Transmission Control Protocol）🔥 🔥 提供面向连接、端到端的**可靠字节流**通信：
- **面向连接**：通信前必须通过三次握手建立连接，结束后四次挥手释放。
- **完全可靠传输**：确认机制 + 超时重传 + 序号重排——保证无差错、不丢失、不重复、按序到达。
- **全双工通信**：TCP 连接双方可以同时收发数据。
- **面向字节流**：应用层发来的数据不被分割为报文边界——TCP 以连续的字节流发送和接收——需要应用自行划分消息边界。
- **一对**一连接，不支持广播多播。

### TCP 报文段格式

TCP 报文段首部 🔥（必记的字段结构和作用）：

| 字段 | 位数 | 作用 |
|------|:--:|------|
| 源端口 / 目的端口 | 各 16 | 发方和收方的进程标识 |
| **序号（seq）** | 32 | 本报文段数据部分第一个字节在当前对方发送流中的字节编号 |
| **确认号（ack）** | 32 | 期望收到对方下一个报文段的 seq（= 已成功接收到的最后字节 seq+1） |
| 数据偏移 | 4 | TCP 首部的长度（以 4 字节为单位）|
| 保留位 | 6 | 保留 |
| **标志位** 🔥 | 6 | URG（紧急）/ **ACK**（确认位有效）/ PSH（推送）/ **RST**（强制复位）/ **SYN**（连接同步）/ **FIN**（释放连接） |
| **窗口（rwnd）** | 16 | 向对方通告接收缓存还剩余多少空间——用于流量控制 |
| 校验和 | 16 | 首部 + 数据的差错检验 |
| 紧急指针 | 16 | URG=1 时指向紧急数据最后一字节的偏移 |
| 选项 | 可变 | 如最大报文段长度 MSS 协商 |

### TCP 连接管理

#### 三次握手

TCP 连接建立需**三次握手** 🔥 🔥（408 年年考seq序列分析）：

1. **SYN = 1, seq = x** （客户端随机选择 x）——客户端请求同步，进入 SYN_SENT 状态。
2. **SYN = 1, ACK = 1, seq = y, ack = x+1** ——服务器分配缓冲区和变量进入 SYN_RCVD 回传同意连接及对 x 的确认，同时发出自己的序号 y。
3. **ACK = 1, seq = x+1, ack = y+1** ——客户端对服务器的序号 y 确认。连接状态变为 ESTABLISHED。

> 为什么是三次而不是两次？为了防止网络中滞留的旧 SYN 分组导致服务器开销建立空连接——第三次握手让客户端承诺该连接确实自己的。

#### 四次挥手

TCP 的释放需要**四次挥手**（因为 TCP 的全双工需要各方向单独关闭）：

1. **FIN = 1, seq = u** 🔥 客户端主动发送连接释放报文——进入 FIN_WAIT_1 状态。
2. **ACK = 1, seq = v, ack = u+1** ——服务器确认收到结束请求。客户端 → 服务器的方向关闭，服务器仍可发剩余数据它方连接。客户端进入 FIN_WAIT_2。
3. **FIN = 1, ACK = 1, seq = w, ack = u+1** ——服务器也决定释放再发自己的 FIN。进入 LAST_ACK 状态。
4. **ACK = 1, seq = u+1, ack = w+1** ——客户端回确认后进入 TIME_WAIT 状态（等待 2MSL 确保服务器收到最后的 ACK 后再正式关闭）——服务器收到后 CLOSED。

> 408 考了又变的经典选择：TIME_WAIT 时长=2MSL（最高段生存时间 × 2）——目的是确保最后发出的 ACK 如果丢失服务器会重新 FIN 而客户端有足够时间再应答。

### TCP 可靠传输

TCP 通过四项核心机制实现可靠性 🔥：

- **序号（Sequence Number）**：将为每一个传送的字节编号——接收方以序号重排数据和检测缺失。
- **确认（累积确认 ACK）**：ACK 号代表此序号以前的数据全部正确收到——是 TCP 标准确认方式（非逐字节 ACK）。
- **超时重传**：每发一个报文段启动一个重传计时器，超时未收到 ACK→ 重发。超时时间动态调整以匹配当前网络 RTT——算法初值为平滑 RTT 估计。
- **冗余 ACK 和快速重传** 🔥：接收方每收到乱序报文段就会立即发出一个重复的 ACK（期望丢失的 seq 的同一确认）——当发送方收到 3 个连续相同的冗余 ACK（额外的三次提醒），不等超时直接重发该丢失报文（快速重传）。

### TCP 流量控制

TCP 用**滑动窗口协议**作流量控制 🔥——接收方在 TCP 首部的**接收窗口（rwnd）** 字段通告自己的接收缓存所乘空间——发送方根据 rwnd 限制发送的数据总量超出接收方能力。当 rwnd = 0——发送方停止发送只定时间隔发送探测报文以检测对方接收窗口重开（零窗口探测）。此滑动窗口机制和链路层的滑动窗口如 GBN/SR 思路相同但针对字节级。

### TCP 拥塞控制

TCP 的拥塞控制 🔥 🔥 是自适应的核心（408 年年考流量图变化曲线）：

**慢开始** 🔥：连接的初始拥塞窗口 cwnd = 1 MSS 慢慢开始。每过一个 RTT，cwnd 倍增（指数增涨）直到达到**慢开始门限 ssthresh** 或检测到丢包。

**拥塞避免** 🔥：cwnd ≥ ssthresh 后进入拥塞避免阶段——每 RTT 将 cwnd 线性增加 1 个 MSS逐一探测可用带宽的边界，安全增长直到丢包。

**快重传** 🔥：收到 3 个重复 ACK 时，判断网络仍有吞吐能力（继续发来重复确认的后续数据），仅就丢的那个段触发重传——不等超时。

**快恢复** 🔥：当发生快重传时，并非回退到慢开始（那太激收），而是将 ssthresh = cwnd/2；cwnd = ssthresh（不回归 1），直接进入拥塞避免阶段——即减速一半再缓慢上涨。

> 🔥 TCP Tahoe 执行快输回启动 → 慢开始（不设快恢复）；TCP Reno（当前最广为部署一致） = 慢开始 + 拥塞避免 + 快重传 + 快恢复 —— 曲线在快速恢复后没回到慢开始那样跌平。

### TCP 计时器

- **重传计时器** 🔥：超时重传的最核心计时器。
- **持续计时器**：rwnd=0 零窗口下仍需定期检测对方是否已重新开启窗口。
- **保活计时器**：长时间空闲的连接发送探测确认对方是否还存活（从崩溃恢复）。
- **TIME_WAIT 计时器**：主动关闭连接一方在最后 ACK 发送后等待 2MSL。

## 常见题型

- **TCP 连接建立和释放的序号分析（大题高频重中之重）**：给三次握手四次挥手的 seq 和 ack 数值，计算每阶段传递的字节数和下一个期望值。
- **拥塞控制窗口变化曲线图**：给吞吐阶段切换图——标记慢开始 ssthresh 位置、拥塞避免平稳升段、快重传后快恢复窗口的变化数值。
- **TCP 往返时间和超时时间计算**：给一系列 RTT 样本，根据公式得到平滑 RTT 估计并导出超时间隔。
- **UDP 校验和计算**：计算 UDP 伪首部 + UDP 首部 + 数据段按 16 位二进制反码求和的结果。

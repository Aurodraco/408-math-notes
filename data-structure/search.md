# 查找

> **考情提示**：查找是数据结构中公式计算密度最高的一章——ASL 的计算、折半查找判定树、BST 删除、AVL 调整，全是高频考点。本章通常 1 道选择题（2 分）+ 偶尔考 AVL 调整大题。核心难点在 AVL 和 B 树——后面有专门篇幅。

## 查找的基本概念

**查找表**是由同一类型数据元素构成的集合，**关键字**是数据元素中某个可以唯一标识该元素的值。查找就是根据给定的关键字值在查找表中找出能匹配的某个记录（如果存在则返回位置，否则返回查找失败的标志）。查找的两个核心操作是**查找**本身和**插入/删除**——只做查找的表叫**静态查找表**（顺序查找、折半查找），需要频繁动态修改的表叫**动态查找表**（BST、AVL、B 树）。

**平均查找长度（ASL）** 🔥 是整个查找章节的核心度量。它定义为：在查找过程中，为确定记录在表中的位置所进行的**关键字的平均比较次数**。公式很简单—— ASL = Σ Pᵢ × Cᵢ，其中 Pᵢ 是查找第 i 个记录的概率（通常假设等概率），Cᵢ 是找到它所需的比较次数。考试中分「查找成功」和「查找失败」两种 ASL——看题干什么：问平均找到目标要比较几次，还是平均确定目标不存在要比较几次。前者用成功的 ASL 公式，后者用失败的。

> 408 选择题最爱考的就是 ASL 的数字计算，绝大部分是等概率假设。如果题给概率不等，按 Pᵢ × Cᵢ 逐项求和即可。

## 顺序查找和折半查找

### 顺序查找

顺序查找（线性查找）是最原始也最通用的查找方式——从头到尾扫描一遍，每个元素都比对一次。它**不要求表有序**，对任何线性表都适用。查找成功的 ASL = (n + 1) / 2（等概率时），查找失败则需要比较 n 次（无序表）或 n+1 次（有序表，可以在遇到一个比目标更大的元素时提前终止）。时间复杂度 O(n)。

**有序表的顺序查找** 🔥：如果表已经按关键字有序排列，查找失败时可以早停——当扫描到一个比目标更大（升序表）的元素，后面的肯定也都比目标大，因此可以直接返回失败。这种情况下失败的 ASL = n/2 + n/(n+1)（近似 n/2），明显优于无序表的 n 次。

> 💀 一般情况下顺序查找不会单独出大题——它通常作为衡量其他高效查找算法的基准（与折半查找、散列表对比 ASL 数值）。

### 折半查找（二分查找）

折半查找 🔥 要求查找表是**有序顺序表**（两个关键词缺一不可——必须有序且必须顺序存储）。每次取表中间位置的关键字与目标比较：相等则命中，小于目标则去右半边继续搜，大于目标则去左半边。每次比较把搜索范围砍半，每轮更新 low/high/mid 三个指针。

折半查找的 ASL 是通过**判定树**来计算的——判定树是一棵描述折半查找所有可能路径的二叉树。n 个元素的判定树深度 = ⌊log₂n⌋ + 1，成功的 ASL ≈ log₂(n+1) - 1 ≈ O(log n)。失败的 ASL 就是对判定树中所有外结点（即查找不成功的结束位置）做平均。

> 折半查找的判定树选择题常考：给 n 个元素，问判定树长什么样——注意它不是完全的二叉排序树，因为 mid 的计算方式是 `(low + high) / 2` 取 ⌊ / ⌉ 会影响树的形状。

> ⚠️ 为什么折半查找对链表无效？因为链表不能 O(1) 直接跳转到 mid 位置。必须顺链走到 mid——这一步就已经是 O(n)，砍半的高效性全然浪费。

### 分块查找（索引顺序查找）

分块查找 🔥 是顺序查找和折半查找的折中方案。将表分成若干逻辑块，块内元素可以是无序的，但块之间必须有序（第一块所有最大关键字 < 第二块所有最小关键字）。额外建一个索引表记录每块的最大关键字及其起始位置。查找时先在索引表中确定目标落在哪个块（折半查找 O(log b) 或顺序查找 O(b)），然后在该块内做顺序查找 O(s)。

块大小 s 怎么选是全章唯一的优化问题：若索引表用顺序查找，总 ASL = (b+1)/2 + (s+1)/2 = (n/s + s)/2 + 1，在 s = √n 时取最小值 √n + 1。若索引表用折半查找则总 ASL ≈ log₂(n/s+1) + (s+1)/2。

> 💀 分块查找在 408 中考的频率很低——过去 10 年只出现过一两次。需要知道概念和等分块下的复杂度计算公式，但不必花大量时间。

## 树形查找

### 二叉排序树（BST）

**二叉排序树（BST / 二叉査找树）** 🔥 的定义：要么为空树，要么满足——左子树上所有结点的值 < 根的值 < 右子树上所有结点的值，且左右子树本身也是 BST。这是考研树形结构的入门式动态查找表。

**BST 的查找**：从根开始，如果目标等于根则命中，小于根去左子树找，大于根去右子树找。完全平衡的 BST 查找复杂度为 O(log n)，退化成一条链时（逐次插入有序数据）复杂度恶化到 O(n)。这就是后续 AVL 树要解决的问题。

**BST 的插入**很简单：从根开始沿着查找路径一直走，走到空位置（NULL）就是新结点的插入点。新增的结点永远是一个叶子，不破坏已有的 BST 结构。

**BST 的删除** 🔥 是三个操作中最复杂的一个，分三种情况：
1. **被删结点是叶子**：直接删除，改动父结点对应的孩子指针即可。
2. **被删结点只有一个孩子**：用这个孩子代替被删结点的位置。
3. **被删结点有两个孩子**：找它的中序前驱（左子树中最右下边的结点）或中序后继（右子树中最左下边的结点），用这个前驱/后继的值覆盖被删结点，然后把前驱/后继（必然是情况 1 或 2）删除。常见做法是用中序后继替换，代码写起来清晰。

> BST 的 ASL 分析：平均 O(log n) 最好 O(log n) 最差 O(n)，选择/简答题问「在什么插入次序下 BST 退化为单链表」答案是**有序插入**。

### 平衡二叉树（AVL）

平衡二叉树 🔥 是 BST 的升级版，核心要求是：**任意结点的左右子树高度差的绝对值不超过 1**。这个高度差就是**平衡因子**——左子树高度减去右子树高度，取值只能为 -1、0 或 1。AVL 保证树的深度始终维持在 O(log n)，从而 BST 最差情况 O(n) 不存在的。

AVL 的插入可能破坏平衡性，需要通过旋转来恢复。标准旋转只有四种（408 年年必考的明星操作）：

| 类型 | 失衡条件 | 旋转操作 | 口诀 |
|------|----------|----------|------|
| **LL** | A 的左孩子的左子树插入导致失衡 | A 右旋（顺转） | 「左左，向右转」 |
| **RR** | A 的右孩子的右子树插入导致失衡 | A 左旋（逆转） | 「右右，向左转」 |
| **LR** | A 的左孩子的右子树插入导致失衡 | 先左旋 A 的左孩子、再右旋 A | 「左右，转两次」 |
| **RL** | A 的右孩子的左子树插入导致失衡 | 先右旋 A 的右孩子、再左旋 A | 「右左，转两次」 |

AVL 的删除同样可能破坏平衡，处理比插入复杂——删除后顺着被删结点的路径一路往上检查每个祖先的平衡因子，失衡就旋转。旋转完还要继续向上检查（因为旋转可能引发祖先失衡）。**408 的大题中 AVL 调整是最常画的推理题**：给一棵 AVL 树和一个新插入的值，要求画出调整后的树。

> 🔥 AVL 被广泛应用于需要频繁有序查找和修改的场景，如数据库索引（虽然更多用 B 树）、平衡二叉树集合（C++ `std::map`、Java `TreeMap` 基于红黑树，但原理相通）。

### 红黑树

红黑树 ⚠️ 在考研中属于「了解级别」——知道定义和性质即可，从未出现过代码题和高难推理。红黑树的五个性质：① 每个结点是红或黑 ② 根是黑 ③ 叶子（NULL）是黑 ④ 红结点的两个孩子必须是黑（没有连续的红） ⑤ 从任意结点出发，每条到叶子的路径上黑色结点数量相同（黑高相等）。

红黑树插入的核心是「染色 + 旋转」——新结点默认红色，如果违反原则四（连续红），根据叔叔结点的颜色分三种情况处理：叔叔红（染色）、叔叔黑且当前是外侧（旋转 + 染色）、叔叔黑且当前是内侧（双旋转 + 染色）。红黑树的删除更复杂，考研不涉及。

> 🔥 红黑树是 Linux 内核调度器、C++ STL `std::map`、Java `HashMap` 桶退化时的后备结构——它是工业界应用最广泛的自平衡二叉查找树，但 408 考得极少，性价比低，不建议深海投入。

## B 树和 B+ 树

### B 树

B 树 🔥 是为磁盘 I/O 设计的自平衡多路查找树。一棵 m 阶 B 树满足：每个结点至多有 m 棵子树（即 m-1 个关键字）、至多有 m 个孩子；根结点至少有 2 棵子树（除非整个树只有根本身）；非终端结点至少有 ⌈m/2⌉ 棵子树（即 ⌈m/2⌉-1 个关键字）；所有叶子在同一层（平衡的）。这保证了 B 树高度始终是 O(logₘ n)，每一层能放大量关键字——所以磁盘 I/O 次数就是树的高度。

B 树的**插入**可能引发结点分裂：一个结点的关键字满 m-1 个后无法再放，就取中间关键字上移到父结点，两边裂开成两个半满的新结点。分裂可能一直传导到根——根分裂会使树高加一。B 树的**删除**可能引发合并：一个结点删除后如果关键字数量低于 ⌈m/2⌉-1，就和相邻兄弟借一个关键字（借不到就和兄弟连同父结点中的分隔关键字一起合并）。

> B 树的分裂与合并是 408 手动模拟大题的重要素材——给一棵 B 树，依次插入/删除若干个值，要求画出每步操作后的 B 树结构。务必手动画过两三次。

### B+ 树

B+ 树 🔥 是 B 树的变形，也是实际数据库索引的默认结构（MySQL InnoDB、PostgreSQL 都用 B+ 树）。与 B 树的关键区别：
- 非叶结点只存放索引（不存实际记录指针），所有实际关键字都集中在叶子层
- 叶结点按关键字从小到大顺序用链表串起来，支持高效范围查询
- 查找任何关键字必须走到叶子层（B 树可能在中间层命中就直接返回）

> 🔥 B+ 树的普及程度远胜 B 树——几乎所有关系型数据库索引都是 B+ 树。408 考研中 B+ 树的考察深度低于 B 树，掌握对比表和叶链表的特点即可。

## 散列表

### 散列表的基本概念

散列表（哈希表）🔥 通过**散列函数**把关键字直接映射到存储位置，理论上查找时间可以接近 O(1)。**冲突**是指两个不同的关键字被散列函数映射到相同的位置——这是不可避免的（鸽巢原理）。衡量散列表拥挤程度的关键指标是**装填因子 α**——α = 表中记录数 / 表长。α 越大，空间利用率越髙，但冲突概率也越大，查找效率下降。

### 散列函数的构造方法

考研只需了解以下四种构造方法 ：

- **直接定址法**：取关键字或关键字的某个线性函数值作为散列地址——最简单，不产生冲突，但对关键字分布要求极高。💀 实际基本不用。
- **除留余数法** 🔥：`H(key) = key mod p`，其中 p 取不大于表长的最大质数。这是最常用也最普适的方法，考研中散列函数默认就是除留余数法。
- **数字分析法**：取关键字中随机性较好的若干位作为散列地址。适合关键字较长且多位的场景 。
- **平方取中法** 💀：计算关键字的平方，取中间几位作为散列地址。历史方法，考研偶尔出现于选项。

### 处理冲突的方法

**开放定址法** 🔥：冲突发生后探测表中下一个空闲位置存入。根据探测策略不同细分为：
- **线性探测**：`Hᵢ = (H₀ + i) % m`，最简单的探测方法，但容易产生「一次群聚」现象——第一处已占的探测会直接堵塞后面一串位置，导致连续的探测链越来越长 ASL 暴增。
- **平方探测（二次探测）**：`Hᵢ = (H₀ ± i²) % m`，探测步长按平方递增（1, -1, 4, -4, 9, -9…），明显缓解群聚问题。表长必须取 4k+3 形的质数才能覆盖整个表。
- **再散列法**：准备第二个散列函数，冲突时调用第二个散列函数再算一个新地址。
- **伪随机序列**：准备一个伪随机序列，冲突时按序列依次探测新地址。

**拉链法（链地址法）** 🔥：不原地换位，而是把冲突的同义词链成一个链表，散列表存放链表头指针。拉链法没有溢出问题（除非内存耗尽），对表长不严格约束，装填因子 α > 1 也能正常工作。ASL 计算时注意区分「成功」和「失败」——成功是遍历链表直到找到目标所需的平均比较次数，失败是遍历所有同义词结点到空的平均次数。

### 散列查找的性能分析

散列查找的 ASL 完全取决于装填因子 α，与记录总数 n 本身无关——这是散列表和之前所有查找方式最本质的区别。开放定址法下的查找成功 ASL ≈ (1 + 1/(1-α)) / 2，拉链法下的查找成功 ASL ≈ 1 + α/2。**考试只要求能对给定表手工计算 ASL**：给定一组关键字、散列函数和冲突处理方法，逐一插入，然后分别计算成功和失败的 ASL（一般用「比较次数」累加取值除以记录数 / 表长）。

> 408 考卷中散列表一类的题出现频率极高——通常一道选择题（2 分）。线性探测和拉链法这两种最主要方法轮流考察。必须训练手工计算 ASL 到熟练无误——它每步都可能犯错（特别是失败的 ASL 计算，经常忘了计算到空地址那一次比较）。

## 常见题型

- **折半查找判定树**：给 n 个元素，画出判定树，标注 find 路线的比较次数。
- **AVL 树的调整**（大题重中之重）：给一棵 AVL 插入/删除，画图确认哪层失衡、怎样旋转。四种旋转分别处理已经达到肌肉记忆才算过关。
- **B 树的插入/删除过程**：每年必有一道选择题或者大题中一部分，考察关键字的移动和结点的分裂/合并。
- **散列表的 ASL 计算**（成功/失败）：线性探测与拉链法交替出现，务必区分成功 vs 失败的对比次数定义。

# 计算机的性能指标与评测

> **考情提示**：性能评测是组成原理的最后一章内容较少，408 中通常 1 道选择题（2 分）。重点：CPU 性能公式、Amdahl 定律计算、三种平均评测方法的理解。知识点精而简，性价比很高。

## 综合性能指标

### CPU 性能公式

CPU 执行程序的整体时间可以由三个因素的乘积精确描述 🔥：

> **CPU 执行时间 = 指令数（IC）× CPI × 时钟周期（Cycle Time）**

或者等价地：

> **CPU 执行时间 = (指令数 × CPI) / 主频**

这三个因子各自受不同的技术和设计决策影响：
- **指令数（IC）**：由编译器、算法和指令集架构决定。RISC 的指令数往往多于 CISC 来完成同样工作，但 RISC 每条指令更简单使得 CPI 更低——这是个互相制衡的关系。
- **CPI（平均每指令执行时钟数）**：由微架构和指令混合决定。流水线、Cache 命中率超标量并行都降低 CPI。
- **时钟周期**：由半导体工艺和微架构的级内关键路径长短决定。单级逻辑越浅时钟可以越高——用硬布线控制的设计通常时钟频率比平微程序更高。

> 408 选择题最爱做的事：给你三个参数，让你算执行时间，或者给你修改一个参数，让你算加速比。

### Amdahl 定律

Amdahl 定律 🔥 是评价系统某项改进对整体性能提升效果的通用原理。如果一项改进可以使系统中一个部分 f（0 < f < 1，程序执行时间中受影响部分的占比）加速 n 倍，整个系统的加速比为：

> **S = 1 / ((1 − f) + f/n)**

当 n → ∞ 时（部分优化趋于无限快），加速比的极限等于 S_max = 1 / (1 − f)。这意味着**系统中未优化的部分的占比立即决定了最大可达到的加速上限**——无论你如何加速另一部分，都无法超越 1/(1-f)。

适用于推演 CPU 提升中某个指令速度、或某个执行阶段（如把某类指令执行时间缩短）带来的整体改善。也广泛应用于系统中互斥改进的综合效果评估——多个改进依次叠加（叠加改进的加速积）的计算需考虑改进间独立性。

> 408 Amdahl 定律是计算题的常年素材——给定 f 和 n 求加速比，或给定一个加速目标和总加速比反过来算需 f 达到多少。公式代入出结果题。

## 基准测试程序

### 测试程序分类

计算机的性能无法用单一指标衡量，需要靠一系列基准程序的实测结果来评估。基准程序分为三级：
- **真实程序** 🔥：实际的应用程序（如 SPEC 所用的程序选择组——包括编译、物理模拟、AI 推理等重量级实际负载）。最能反映实际价值，但运行一次费时极长。
- **核心程序（Kernel Benchmarks）** ⚠️：从实际程序中提取出的计算密集部分（矩阵乘法、FFT 等），运行时间可控，同时反映局部性能。
- **合成测试程序** 💀：纯粹人工构造的无实际应用对应的测试——如 Dhrystone、Whetstone——目前已被认为不能反映真实性能，不再被用于真正评测。

### SPEC 基准测试

SPEC（Standard Performance Evaluation Corporation）🔥 是产业和学术界广泛使用的标准化性能评测系统。SPEC CPU 以真实程序为基础，每个基准程序代表某一类高性能计算或通用计算的典型工作负载。SPEC 的结果用多个程序成绩的几何平均得出综合评价——避免单一极端成绩歪曲总体。

### 综合性能度量

- **MIPS（每秒百万条指令）** 🔥 = 指令数 / (执行时间 × 10⁶) = 主频 / (CPI × 10⁶)。MIPS 越高，表面速度越快——但**无法跨指令集对比**（RISC 的一条指令 ≠ CISC 的一条指令）。常见选择题陷阱：给一个改善增加了 MIPS 值但实际执行时间反而更长了——发生情况是 CPI 上升但指令数下降不同步。
- **MFLOPS / GFLOPS（百万/十亿次浮点运算每秒）** 🔥：专用于衡量科学计算和浮点密集负载的性能。比 MIPS 较为可跨机器对比——因为浮点操作的功能在给定精度下是等效的。
- **这些指标的局限性**：MIPS 不能跨架构对比、不能反映 I/O 影响；MFLOPS 只反映浮点部分，不对标整数和内存操作。

## 影响性能的因素

### 硬件因素

- **时钟频率**：更高的频率直接缩短每条指令耗时——受 CPU 能耗墙和散热限制。
- **Cache 配置**：Cache 容量、相联度和层次数目严重影响平均访存延迟。L1 命中率比任何单一参数影响更大。
- **流水线深度**：更深化流水线提升吞吐量，但加深分支错判的惩罚，存在收益递减。
- **总线带宽**：决定 CPU 与主存/外设的数据供给速率。总线瓶颈会导致 CPU 吃饱但饿死的问题——空有高速没有足够数据搬运。

### 软件因素

- **编译器优化**：通过指令调度、循环展开、内联函数、消除冗余引用等方法大幅降低指令数并提高指令级并行度。
- **指令集**：RISC 编译器优化空间比 CISC 更大（因为结构越简单越容易优化映射）。选择合适的指令集减少 CPI。
- **算法复杂度**：数据结构的最重要的效率维度——一个 O(n²) 的代码无论用多快的 CPU 都无法超越 O(n log n)。

## 性能评测方法

评测多组基准测试程序时，用哪种平均方式很重要：

- **算术平均**：各程序执行时间直接求平均——对所有程序等权重。当各程序执行时间尺度一致时用算术平均。
- **几何平均** 🔥：对 n 个程序的**执行时间比规一化结果**求几何平均——是 SPEC 和大多数标准评测中的规范。相比算术平均，不会因单程序极值扭曲整体的结论。
- **调和平均** ⚠️：用执行速率的倒数平均值再倒数——适用于衡量吞吐率类指标的均值。考研了解公式即可。

## 常见题型

- **Amdahl 定律的应用计算**：给 f 和 n，算 S；或给定目标 S 反算 f。记得 S 永远 ≤ 1/(1−f)。
- **CPU 执行时间的计算（综合题）**：给 IC、CPI 和主频，套公式 `T = IC × CPI / f`。单位要用 M/G/K 谨慎换算（如 1ms = 10⁶ ns）。
- **MIPS 与 CPU 执行时间的换算关系**：一个公式的正反用：`MIPS = f / (CPI × 10⁶)`；`T = IC / (MIPS × 10⁶)`。常考「提升 MIPS 反而降低实际速度」的陷阱辨析。

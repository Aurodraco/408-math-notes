# 内存管理

> **考情提示**：内存管理是操作系统分值最高的章节之一——页式地址变换几乎每年一道大题（≈10 分），页面置换算法缺页率计算也是固定选择题 2 分。本章与组成原理的 Cache/虚拟存储器有大量交叉，学通一科可以互促另一科。

## 内存管理基础

### 内存管理的功能

操作系统内存管理四大核心功能 🔥：
1. **内存空间的分配与回收**：进程创建时分配内存，退出后回收。分配的单位可以是连续分区、页或段。
2. **地址转换**：将程序的逻辑地址转换为对应的物理内存地址——由 MMU（内存管理单元）硬件配合 OS 页表/段表完成。
3. **内存空间的扩充**：通过虚拟存储技术让程序觉得自己拥有比实际物理内存更大的地址空间。
4. **存储保护**：确保每个进程只能访问自己被分配的内存区，不能越界偷窥其他进程的数据——通过基址/限长寄存器或页面保护位实现。

### 程序的装入和链接

程序从源代码变成可在内存中运行的进程，经历两个阶段：

**装入方式** 🔥：
- **绝对装入** 💀：编译时就确定所有内存地址——程序之后只能在预先分配的那些物理地址上运行。早期单道 OS 使用。
- **可重定位装入（静态重定位）** 🔥：程序编译偏移从 0 开始，装入时根据分配的起始物理地址一次性修正所有地址引用——装入后不能再移动。
- **动态运行时装入** 🔥 🔥：程序开始执行后地址转换持续在每次访存时由硬件重定位寄存器完成——程序运行过程中可在内存中搬移（交换）。现代 OS 普遍采用这种方式。

**链接方式** 🔥：
- **静态链接**：所有目标模块+库被编译程序统一链接成一个完整可执行文件——文件大且共享共用库被重复占用。传统 .EXE 编译。
- **装入时动态链接**：先将各模块调到物理内存里再记录符号链接关系——类似共享库的装入时确定。
- **运行时动态链接** 🔥：程序运行到要调用某库时才从磁盘找该库加载到内存并链接。现代 OS 全部负载于此方式——典型的 Windows DLL 和 Linux .so 加载机制。

### 逻辑地址与物理地址

**逻辑地址（虚地址）** 🔥：CPU 产生的地址——每道程序在以 0 为起点的逻辑地址空间中编写。**物理地址**：主存单元的硬地址——逻辑地址在访存前必须转换为物理地址。这个转换过程是组成原理 MMU 与操作系统页表的协作结果，也是后续所有内存管理的讨论中心。

### 内存保护

最简单的保护就是给每个进程一对**基址/限长寄存器**——基址寄存器存该进程分配内存区的启始物理地址，限长寄存器存该内存区的长度。每次进程访问内存，硬件自动检查：`基址 ≤ 访问地址 < 基址 + 限长`，犯界则产生地址越界中断。

> 更细粒度的保护在页式/段式系统中实现——每页/段有保护位（读/写/执行）限制该进程对该页的访问方式。

## 连续分配管理方式

系统将物理内存划分成若干连续的分区，进程占据一整块分区。

### 单一连续分配

单一连续分配 💀 是最原始简单的分配方式——内存只分两道：低地址部分全给 OS，剩余全给一个用户进程使用。每次同时只能有一道用户作业。早期 MS-DOS 就是这种方式。已完全过时。

### 固定分区分配

固定分区 🔥 把用户内存预先划分为若干固定大小的区域——每个区可以装一道作业。分区分大小相等（用完不能再接多作业的浪费）或不等（提供不同大小选择减低浪费）。分区个数和大小不可动态更改——灵活性差。优点是实现极简单。可用**分区使用表**管理各分区占用情况。

> 💀 已在现代系统中几乎消失（嵌入式有使用），408 了解分区等大小的浪费为主。

### 动态分区分配

动态分区 🔥 根据进程实际需要大小实时从空闲内存区划出一块——分区大小、数量完全动态。当进程结束时释放回回收插入空闲区表——此时可能出现大量不连续的小碎片（外碎片）。分配算法：

- **首次适应（First Fit）** 🔥：沿空闲分区链从头找，找到第一个 ≥ 需求的空闲区就分。速度快，查找开销小。但低地址区易累积大量小碎片区——下次分配搜索要经过一串小程序拒绝才找到目标。
- **最佳适应（Best Fit）**：找所有空闲区中大小最适合（刚好不小于需求尺寸且最小的那个）。分配后剩余空闲区最小——看似最佳但其实生成大量极小的无用碎片越来越碎片化查找变慢。
- **最坏适应（Worst Fit）**：每次分配最大的空闲区——剩余空闲区依然较大。看似能避免微碎片但反而导致大空闲区过快缩小而使后续分配大空间请求失败。
- **邻近适应（Next Fit）**：和首次适应一样，但从上一处分配位置的继续往下找——平衡了低地址和高地址碎片的分布。

> 408 经常考试：给当前的空闲区链和分配或释放序列，问每种算法在哪处分配、事后空闲区链的状态变化。

### 可重定位分区分配

可重定位分区 🔥 通过硬件能力动态**紧凑**——将内存中所有运行中的进程移动使其分区聚拢到一端，另一端的空闲区合并成一个大区块供重新使用。但搬动程序开销极大，实际中极少做。多数 OS 用「页式」回避外碎片问题。

### 动态分区分配的回收

释放一个分区后检查它两邻区是否也空闲——若相邻空闲就可合并成一个更大的空闲区以减少碎片。考研里的算法就是边界标记——在每个分区的头和尾保存分区大小和是否空闲的标记以便合并时定位相邻分区。

## 非连续分配管理方式

### 基本分页存储管理

分页存储 🔥 🔥 是现代操作系统内存管理的基石。将物理内存切成大小相等的固定块——**页框（Frame）**，逻辑地址的进程也切成同样大小的**页面（Page）**。进程的页面可以非连续地装入任何一个空闲的页框中，由每个进程的一张**页表**记录映射关系：页号 → 框号。

**地址变换机构** 🔥 🔥：CPU 的虚地址（设为 32 位）分为**页号 p**（高位）和**页内偏移 d**（低位）。MMU 的页表基址寄存器存有页表首地址 → 查到页号 p 对应的框号 f → 物理地址 = f × 页面大小 + d。这个过程可由快表大幅降低因查表需要的一次额外访存。

**快表（TLB）** 🔥 在组成原理已详述——这里再次强调它的位置：TLB 在地址转换的第一步就被查询——命中直接拿到框号不需访问页表；不命中才访问页表并将本次转换的页号-框号对装入 TLB。

**多级页表** 🔥：64 位系统地址空间超出单级页表的表示——采用多级分层页表。每个级别的页表本身也是放在一个页框里的数据——按需建页表，内存只维护真正在使用的那部分页表，省下大量页表预留空间。

**页表项大小与页内偏移的计算** 🔥：408 大题的核心送分计算——给定页面大小（如 4KB = 2^12）、地址长度（32 位），偏移 d 固定 = 12 位（= 页面大小）。页号 p = 32 − 12 = 20 位——所以最多 2^20 页面框，页表最多 1M 项。如果是 64 位系统需要更多层次分解。

### 基本分段存储管理

分段管理 🔥 和页式完全不同——不是等分内存而是按程序的逻辑段分开分配给自然边界（如代码段、数据段、栈段、堆段）。每个段长度不等，有一张段表记录各段的基地址和界限。地址转换涉及段号和段内偏移——段表中需查界限确保访问不越界。

分段对比分页的差异内存：
| 对比 | 分页 | 分段 |
|------|------|------|
| 单位 | 等长页（物理固定） | 不等长段（逻辑意合单位） |
| 地址结构 | 一维——页号+偏移 | 二维——(段号, 段内偏移) |
| 可见性 | 用户不可见（硬件施作） | 用户可见（编译器设段） |
| 碎片 | 内碎片（每程序恰好分页） | 外碎片（段参差破碎） |
| 保护 | 按页保护（粗粒度） | 按段类型保护（自然） |

> 💀 纯粹的分段现在基本不用——x86 的分段已被软化到退化作用，现代系统以页式或多级页表为主。

### 段页式存储管理

段页式 🔥 是分段与分页的结合——先把程序按逻辑分成若干段，每段内部再进行分页。地址变成三维结构：(段号, 段内页号, 页内偏移)。进程获得一个段表指向各段的页表地址，页表再指物理页。段页式的优点有所有：从段层面保护、从页层面无外碎片。缺点：地址转换需要两次查表（段表 + 页表），比纯页式又慢了一步——所以 TLB 也更重要得多。

> 408 对段页式考得很少，停留在画地址结构图和推两级翻译步骤即可。

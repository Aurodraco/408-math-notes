# 进程管理

> **考情提示**：进程管理是操作系统最核心的章节——PV 信号量操作（大题几乎年年必考 10 – 15 分）+ 调度算法计算（选择题高频）+ 银行家算法（选择题/简答题经常出现）。本章是你 408 操作系统得分的基本盘。

## 进程与线程

### 进程的概念和特征

进程 🔥 是程序在给定数据集上的一次执行过程——**系统进行资源分配和调度的独立单位**。进程由三个实体组成：**进程控制块 PCB**（操作系统用来管理进程的数据结构，存放进程标识符、状态、优先级、程序计数器、寄存器映像、内存分配信息、打开文件表等所有进程信息）、**程序段**（指令代码）和**数据段**（运行时变量、堆、栈）。

进程的四大特征（与 OS 四大特征一一对应）：
- **动态性**：进程是程序的一次执行——有创建、活动、暂停和结束的生命周期。
- **并发性**：多个进程可在同一时间间隔内同时运行。
- **独立性**：进程是独立获取资源的实体。
- **异步性**：各个进程按各自的不可预知的速度向前推进。

> 408 核心辨析：**进程与程序的区别**——程序是静态的指令序列（存在磁盘上的可执行文件），进程是动态的执行实例（在内存中有 PCB 的那个）。一个程序可以对应多个进程（如多次打开同一浏览器），一个进程只能执行一个程序。

### 进程的状态与转换

经典**五状态模型** 🔥（408 必须画熟整个状态转换图）：

1. **创建态（New）**：进程被创建尚未加入就绪队列——OS 正在分配 PCB、装入程序、初始化资源。
2. **就绪态（Ready）** 🔥：进程已得到除 CPU 外的所有必要资源，只等调度器分配 CPU。
3. **运行态（Running）** 🔥：进程正占用 CPU 执行指令。单 CPU 系统同时最多一个运行态进程。
4. **阻塞态（Blocked / Waiting）** 🔥：进程因等待某事件（I/O 完成、资源可用、收到信号等）暂时无法继续。
5. **终止态（Terminated）**：进程正常结束或异常终止，OS 正在回收 PCB 和资源。

状态转换路径：创建 → 就绪（创建完成）→ 运行（调度选中）→ 终止（运行结束）。运行 → 就绪（时间片用尽或高优先级抢占）。运行 → 阻塞（请求无法立即可用的资源）。阻塞 → 就绪（等待事件发生）。

**七状态模型**（含挂起）：在五态基础上新加两个挂起态——**就绪挂起**和**阻塞挂起**。当内存不足时，OS 可以将部分进程从内存对换到外存（swap out），此过程就进入挂起态。挂起状态的进程即使满足运行条件也不能到 CPU 上——必须被先换入内存再做调度。这是中级调度的依据。

### 进程控制

进程控制用**原语（Primitive）** 🔥 完成——原语是在内核态下运行的不可中断的最底层过程，通过硬件的关中断/开中断指令保证执行的原子性。核心原语包括：

- **创建原语（Create）**：申请空白 PCB → 为进程分配资源 → 初始化 PCB → 将进程加入就绪队列。
- **撤销原语（Destroy）**：回收进程所有资源 → 释放 PCB → 从队列移除。
- **阻塞原语（Block）**：将当前进程从运行态移入阻塞队列，之后调度下一就绪进程运行。
- **唤醒原语（Wakeup）**：将一个被阻塞进程从阻塞队列移入就绪队列（不立即上 CPU）。
- **切换原语（Dispatch）**：保存当前进程的上下文到 PCB → 从就绪队列中选择一个新进程 → 把其上下文从 PCB 恢复到 CPU 寄存器 → 运行。

> 408 问的最多的一句话就是「进程切换何时不能发生」，答案是操作系统内核在执行临界段关键操作时必须屏蔽中断——此时不能响应调度申请。

### 进程的通信

多个协作进程之间经常需要交换数据或多进程互操作。通信机制有三类：

- **共享存储** 🔥：OS 在内存中分配一个共享空间——多进程将其映射到各自虚拟地址空间中。进程直接通过对空间里的共享变量读写进行通信——同步要由程序员处理。是效率最高的 IPC 方式🔥。
- **消息传递** 🔥：操作系统提供 send/receive 原语——无需共享空间。又可按是否只指定一个接收进程区分为**直接通信**（发送给特定目标进程）和**间接通信**（通过共享的 mailbox/端口发收消息）。微内核系统中这是唯一的 IPC 方式。
- **管道通信（Pipe）** 🔥 🔥：特殊的文件方式——连接一个写进程和一个读进程的 FIFO 缓冲区。管道分无名管道（临时）用于父子进程和命名管道（有名 UNIX 中叫 FIFO）用于任意进程间。UNIX 中的 `|` 是一个管道的典型应用。写满读空时都有一个阻塞的行为。

### 线程

线程 🔥 🔥 是在进程内部的一条执行流——每个进程至少有一个线程。线程共享该进程的所有资源（地址空间、文件、信号等），但各自**独有**：线程 ID、PC、寄存器上下文、栈。线程是**CPU 调度的基本单位**——CPU 不再以进程为调度对象而是以线程。

**线程与进程的核心对比**：
| 对比项 | 进程 | 线程 |
|--------|------|------|
| 调度 | 拥有资源的单位 | 是 CPU 调度单位 |
| 资源所有 | 有独立的地址空间 | 共享所属进程全地址空间 |
| 并发代价 | 创建/切换代价大 | 极轻量——创建和切换开销远低于进程 |
| 隔离性 | 独立内存空间互不干扰 | 同一进程中的线程可互相访问内存数据（协作但易竞争） |
| 系统开销 | 进程切换需要完全切换地址空间 | 同一进程线程切换不需换地址空间 |

**线程的实现分三级**：
- **用户级线程** ⚠️：线程库全在用户态工作，内核完全不知道线程的存在——对内核一个进程只有一个调度单元。切换快但任何一个线程堵塞整个进程被阻塞。 💀 逐渐被内核线程淘汰。
- **内核级线程** 🔥：每个线程都由内核管理——调度、阻塞、切换由内核完成。多核 CPU 上同一进程的内核线程可以真正并发执行。Linux 使用内核线程为主。
- **多线程模型**：**一对一**（一个用户线程映射到一个内核线程），**多对一**（多个用户线程映射到一个内核线程——用户并发但内核串行），**多对多** 🔥（用户线程与内核线程非一一绑定——平衡用户灵活性及内核性能）。现代 OS 多用一模型。

## 处理机调度

### 调度的层次

分层调度从宏观到微观分三级 🔥：

- **高级调度（作业调度 / 长程调度）**：决定将辅存中的哪些作业装入主存进入就绪队列。运行频率极低（数秒到数分钟一次）——批处理操作系统独有，现代个人计算机不设作业调度。
- **中级调度（内存调度）** 🔥：决定哪些进程暂时换出到外存（挂起）以腾出内存。根据内存压力触发——频率远高于高级调度。
- **低级调度（进程调度）** 🔥 🔥：决定就绪队列中哪个进程/线程获得 CPU。运行频率最高——几十毫秒一次。这是考研调度算法的主要考察对象。

### 调度的时机

进程调度的发生时机 🔥：① 当前运行进程主动放弃 CPU（结束或阻塞）② 当前进程时间片用完 ③ 有更高优先级的进程进入就绪状态（可抢占时发生）。**不能进行调度的情形**：CPU 正处理中断时非中断进程不能施实调度；操作系统维护临界区数据一致性时关中断期间必须封锁调度。

### 调度的方式

- **非剥夺式调度（非抢占式）**：一个进程拿到 CPU 后一直运行到自身弃权（终止或阻塞）或时间片自然结束。FCFS、SJF 非抢占变种。优点是无上下文切换抢断的复杂——缺点一旦 CPU 被一长作业霸占，后续短作业必须等到天荒地老。
- **剥夺式调度（可抢占式）** 🔥：操作系统随时可从当前进程中剥夺 CPU 给更高优先级或更符合条件的进程。RR、多级队列、多级反馈队列均为此类。现代所有通用系统均采用可抢占调度。

### 调度算法

408 调度算法的核心要求就是对每种算法会手工计算**平均周转时间**和**平均等待时间** 🔥。

| 算法 | 类型 | 抢占 | 核心特点 |
|------|------|:--:|------|
| **FCFS** | 作业/进程 | 否 | 串行先进先出——长作业友好、短作业被堵死。**
| **SJF / SPF** | 作业/进程 | 均可 | 理论上平均等待时间最短——但谁诚实报告所需运行时间？难以预测。长作业可能饥饿。
| **HRRN（高响应比优先）** 🔥 | 作业 | 否 | 响应比 R =（等待时间 + 所需运行时间）/ 所需运行时间 = 1 + 等待/服务。优先调度最高 R 作业——综合补偿短作业和防长作业饿死。是作业调度优选。**
| **时间片轮转（RR）** 🔥 | 进程 | 是 | 固定时间片轮转。关键在时间片大小选择——太大退回 FCFS 无差别，太小上下文切换开销暴增。**
| **优先级调度** | 进程 | 均可 | 可就优先级比较调度。可能发生的饥饿通过增进等待作业的优先级慢慢补偿（aging）。|
| **多级队列** | 进程 | 是 | 多个就绪队列如前台交互队列高优先级、后台批处理低优先级——队列间指定策略队列内其他算法。|
| **多级反馈队列** 🔥 🔥 | 进程 | 是 | 重要内存最综合的设计——时间片可动态延长：新进程进最高优先级短时间片队列，用满 CPU 时间片后降至少优先长时队列。既保证交互响应快又避免了批处理饥饿。现代 OS 的调度思想重要参考。|

> 对这七种算法 408 重点考：**FCFS / SJF / HRRN / RR的周转/等待时间计算 → 选择题；多级反馈队列的巧妙参数变化—→ 简答题。** 练习题优先反复算上几组数据，掌握套公式即可。

## 进程同步

### 临界资源与临界区

**临界资源** 🔥 是一次只允许一个进程使用的共享资源（如打印机、特定数据结构的公共变量）。**临界区**是进程中访问临界资源的代码段。同步的根本原则就是要确保**同一时刻只有一个进程进入自己的临界区**。

同步机制应满足以下四条准则（408 必背）：
1. **空闲让进**：当没有进程在临界区时，想进入的进程可以立即进入。
2. **忙则等待**：已有进程在临界区时，其他试图进入的进程必须等待。
3. **有限等待**：请求进入的进程必须在有限时间内能进入——避免死等饥饿。
4. **让权等待**：不能进入临界区的进程应立即释放 CPU ——不要原地空转占用 CPU（即「忙等 Busy Wait」）。

### 互斥的实现方法

**软件方法（Peterson 算法）** 🔥 🔥：纯利用两个线程共享的标志变量和 turn 轮转值，实现双进程的严格互斥：每个进程先设置本进程「有意进入」标志，然后把 turn 让给对方，之后一直检查「对方有意进入且 turn 还是对方」——若是则等待，否则进入临界区。Peterson 不需要硬件特殊指令，是在双进程上看就完整的软件互斥。408 偶尔简答题要求写出 Peterson 伪代码。

**硬件方法** 🔥：由硬件提供原子指令实现最简单的互斥操作：
- **关中断**：进入临界区前关中断（禁止响应外中断防止调度），出临界区后开中断。实现极简单，但不适用于多核（关中断只对所在的 CPU 有效），且影响整个系统中断响应。 💀 实际中不用。
- **TestAndSet 指令**：硬件提供的原子测试并置位指令——读取一个 lock 变量的旧值同时将它设为 1。若旧值为 0（锁空闲）则成功获得锁进入临界区；若旧值为 1（已被占）则不断循环 TestAndSet 直到它变为 0——这是「自旋锁」的实现。Spinlock 在锁持有时间极短时非常高效 🔥（多处理器同步中广泛使用）。
- **Swap 指令**：原子交换锁变量和一个临时常量 1 的内容比照同样逻辑实现互斥。

> 408 问硬件指令最核心的一句话是「硬件提供写过程不可分割的检测—置位操作」。选择题/判断就容易在这里考原子性。

### 信号量

信号量 🔥 🔥 是操作系统实现进程同步/互斥的经典最有效的机制。分为两类：
- **整型信号量** 💀：单纯一个整数值，等待时若值 ≤ 0 让进程在该信号量上忙等。不让权等待——浪费 CPU。只在简单场合有使用价值。
- **记录型信号量** 🔥 🔥：结构体包含一个整型值和一个进程的阻塞队列。当进程执行 P 操作发现资源不足 -> 把自己阻塞放到信号量的等待队列里并进入让权等待。V 操作唤醒队列中的等待进程。结构符合全部四个同步原则，是实际 OS 使用的信号量。

**P 操作（wait / 减一操作）** 🔥：
```
void P(semaphore s) {
    s.value--;
    if (s.value < 0)         // 没有可用资源
        block(s.queue);     // 将当前进程加入阻塞队列并放弃 CPU
}
```

**V 操作（signal / 加一操作）** 🔥：
```
void V(semaphore s) {
    s.value++;
    if (s.value <= 0)       // 有人在等待
        wakeup(s.queue);    // 从阻塞队列中唤醒一个
}
```

> 注意：value 的初始值 = 可用资源数。value 为负时，绝对值代表等待该信号量的进程数量。若 value >= 0 则表示当前能提供多少无人等待的资源。408 几乎所有 PV 大题都是基于记录型信号量。

### 经典同步问题

408 必考的经典同步问题（PV 大题几十年来源集中于此）：

**生产者-消费者问题** 🔥 🔥（最基础最常考）：一组生产者线程生成数据放入有界缓冲区，一组消费者从缓冲区取出数据——缓冲区是共享的临界资源。解法：用一个**互斥信号量 mutex**确保缓冲区的互斥访问；用两个同步信号量 `empty`（初始 = n，表示空位数）和 `full`（初始 = 0，表示满数据量）。生产者先 P(empty) 再 P(mutex) → 放数据 → V(mutex) → V(full)；消费者先 P(full) 再 P(mutex) → 取数据 → V(mutex) → V(empty)。同步信号量放在互斥信号量前面是必须的——防止死锁。

**读者-写者问题** 🔥 🔥：共享资源的多个读者可以并行读，写者写入时必须独占。经典解法：引入一个 `readcount`（当前在读读者数），第一读者进时 P 写信号量以阻止写者，最后读者出时 V 写信号量让写者有机会。写者只需直接 P 写信号量执行写写入后 V。注意 readcount 也要用互斥信号量保护。

**哲学家进餐问题** ⚠️：五个哲学家围坐在五个筷子之间——每人需要同时拿到左右两根筷子才能进餐。经典问题展示了无良好同步设计中可能出现的循环等待死锁。解法包括限制同时进餐者数量、或要求先拿左筷再试右筷且拿不到则放下（但此举引入饥饿风险）。

**吸烟者问题** ⚠️：一个供应者和三个吸烟者的多重生产-消费的小扩展——供应者异步随机提供两种材料，只有拥有第三种配套材料的吸烟者才能合成并吸烟。该题旨在巩固 PV 信号量的逻辑推演，408 出镜少于前面两个。

> 408 大题近年出现的趋势是直接用生产者-消费者或读者-写者作为模板，在一段伪代码或代码中让你补完信号量和 PV 位置。这类题练上五道就能形成反射。

### 管程

管程（Monitor）🔥 是为高级语言设计的一种同步抽象——它把共享数据全包装在一个结构内，由编译器确保每次只允许一个进程进入管程（自动互斥）。管程提供**条件变量**（condition），与条件变量关联的操作是 `wait()`（使进程在条件变量上阻塞等待）和 `signal()`（唤醒一个在该条件变量上等待的进程）。

管程减少了程序员显式使用 PV 信号量的心智负担。管程由开发方引入后，很多现代语言都借鉴了管程思想（如 Java 的 `synchronized` 块和 `wait()/notify()`）。

## 死锁

### 死锁的定义和必要条件

死锁 🔥 🔥 是多个进程因争夺资源而形成的互相等待僵局——每个进程都占用了对方需要的资源但自己在等待对方释放另一个资源，无外力推移下永远无法继续。

死锁的四个必要条件（必须同时满足才会死锁）🔥：
1. **互斥条件**：资源只能被一个进程独占。
2. **不剥夺条件**：进程获得的资源不能被强行夺取，只能由自己释放。
3. **请求和保持条件**：进程已经保持了至少一个资源，还要再请求新的被其它进程占用的资源。
4. **循环等待条件**：存在一个进程集合 {P₁, P₂, ..., Pₙ}——P₁ 等 P₂ 占的资源，P₂ 等 P₃，...，Pₙ 等 P₁。

### 死锁预防

通过破坏四个必要条件之一达到预防效果：
- 破坏**互斥**：非可用 ❌——很多资源天生只能互斥。
- 破坏**不剥夺** 🔥：若进程请求的资源不可用，操作系统强制剥夺它已经占用的资源——代价极大，很少使用。
- 破坏**请求和保持** 🔥：要求进程一次性申请所有需要的资源——开始前全部分配或全不分配（静态分配）。简单但资源利用率极低（请求全部再工作，很多资源几乎不用但被占住）。
- 破坏**循环等待** 🔥：对资源类型分配全局编号——进程只能按编号顺序申请资源，低号先申请，不会回头申请高号再去等低号，杜绝环路。实际中常用来预防死锁。

### 死锁避免

死锁避免 🔥 不破坏死锁必要条件而是判断每步分配如果进行下去是否会导致死锁。最著名的是**银行家算法** 🔥 🔥。这个算法模仿银行贷款的风险管控——系统在每次进程请求资源时先模拟分配——如果模拟后系统还在**安全状态**（存在一种进程执行顺序能让所有进程依次完成），则批准分配；否则延迟分配让请求进程等待。

安全状态判定方法：找出一个安全序列 {P₁, P₂, ...} ——顺着序列每个进程都可用当前可用资源加上前面进程释放的资源完成。如果没有安全序列存在——系统处于不安全状态，可能死锁。

408 银行家算法的考题统一套路：给一个资源表（max/alloc/need）和当前 available；问是否安全→ 找出安全序列 → 若某进程发出新请求，可否立即分配？答题步骤严密 — 先计算 Need = Max − Alloc，然后按 Need 每次同 available 比较找满足条件的进程连成一个序列。

### 死锁的检测和解除

**资源分配图** 🔥：用进程结点（圆）和资源结点（方框）以及申请边与分配边构成有向图。若图中无环——无死锁；若图中每条资源方框中存在圈——每个资源只有一个实例时为死锁充分条件；若资源类型多个实例可能存在环但无死锁（环外有进程能释放共享资源化解环内堵塞）。

**死锁定理**：通过不断释放所有不被阻塞的进程（占有资源的进程，且与等资源的边可以消去）的分配边以达到化简分配图，化简完仍存在边未消——死锁存在。

**解除死锁的两种方法** 🔥：
- **撤销进程**：撤销一个或多个处于死锁中的进程释放占用的资源。选择牺牲进程的依据有优先级、运行时间、使用资源量等。
- **剥夺资源**：从某些进程手中剥夺资源分给其他进程使死锁解开。实现极其困难。

## 常见题型

- **PV 操作解决同步互斥（大题核心）**：题型基本出在生产者-消费者/读者-写者两大模板，要求补充信号量初始化及过程代码中 PV 位置。重点考点是互斥信号量的 P/V 放在同步信号量内部、多个资源条件组合 PV 的逻辑顺序，以及边界条件（是否需要外面加锁）。
- **调度算法的计算**（选择题高频）：给作业到达时间和 CPU 时间，手动用 FCFS/SJF 非抢占/HRRN/RR 各自计算平均周转时间和平均等待时间。列阶段表逐一加和。
- **银行家算法的安全性检查**（简答/选择往复考）：推推安全序列→ 分配判断→ 给出列表位。
- **死锁的判定**：判断给定进程和资源的资源分配图是否有死锁、举出化简的步骤。
